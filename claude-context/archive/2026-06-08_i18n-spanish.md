# Multi-Language Support (i18n) ŌĆö Spanish

**Completed**: 2026-06-08
**Status**: Shipped to production

## What Was Built
Implemented Spanish as first non-English language with full i18n infrastructure. Backend stays English-only (DB, Qdrant, APIs unchanged) ŌĆö translation happens at two boundaries: the LLM synthesizer streams responses in the target language via `LANGUAGE_INSTRUCTION`, and the React frontend localizes ~600+ static UI strings via `react-i18next` with bundled JSON catalogs. Language selector (globe icon) in splash and workspace headers. All chat components, landing page, map UI, sidebar data cards, term definitions, and pages (Pricing, Scorecard, Explorer) fully localized. Conversation language persisted in DB. Adding a new language requires only locale JSON files + a LanguageSelector option.

### Phase 2: Backend ŌĆö Prompt & Synthesis Changes

#### 2.1 Conversation synthesis (`backend/conversation.py:146`)

### Phase 3: Backend ŌĆö Translation Endpoint

#### 3.1 New module (`backend/translate.py`)

```python
async def translate_text(text: str, target_language: str, request_group: str = "") -> str:
```

- Uses Haiku via `tracked_create()` with system prompt: "Translate the following text into {language_name}. Preserve all legal references, section numbers (┬¦), formatting, and proper nouns. Return only the translated text."
- Cache layer: `TTLCache(ttl_seconds=86400, maxsize=4096, name="translation")` keyed by `sha256(text + target_language)`
- Used for: (1) on-demand source chunk translation from the frontend, (2) clarification message translation

#### 3.2 API endpoint (`backend/main.py`)

```python
class TranslateRequest(BaseModel):
    text: str = Field(max_length=10000)
    target_language: Literal["es", "pl", "zh-CN", "zh-TW"]

@app.post("/api/translate")
async def translate_endpoint(req: TranslateRequest, ...) -> dict:
    translated = await translate_text(req.text, req.target_language, ...)
    return {"translated": translated, "language": req.target_language}
```

Rate-limited with existing `check_rate_limit()`.

### Latency Impact

| Path | Added Latency | Notes |
|------|--------------|-------|
| English (default) | **+0ms** | No changes to any LLM call or prompt |
| Non-English, first message | **+0ms** | No conversation synthesis needed (no history) |
| Non-English, follow-up | **+300-600ms** | Forced conversation synthesis via Haiku (bypasses English heuristics) |
| Non-English, synthesis | **+100-300ms** | Language instruction adds ~100 tokens to system prompt; non-Latin tokenization ~10-20% slower |
| On-demand source translation | **+800-1500ms** | User-initiated, cached, does not block chat |

Total per-query overhead: ~200-600ms for non-English follow-ups, ~100-300ms for first messages. Typical queries go from 3-8s to 3.5-8.5s.

Prompt caching: Separate cache entries per language. English cache (majority of traffic) completely unaffected.

### Risks & Mitigations

1. **Citation marker corruption** ŌĆö Synthesizer might translate `[data:crime]` or `[1]`. Mitigation: explicit preservation instructions in LANGUAGE_INSTRUCTION + eval test cases.
2. **Chinese tokenization cost** ŌĆö Chinese text uses ~15-25% more tokens. Mitigation: monitor via admin dashboard; absolute cost still small (~$0.01/query).
3. **Conversation synthesis false negatives** ŌĆö English-based `needs_synthesis()` heuristics won't match non-English pronouns/patterns. Mitigation: force synthesis for all non-English follow-ups.
4. **Term definition translation quality** ŌĆö Zoning/TIF/ARO are Chicago-specific. Mitigation: keep official program names in English per LANGUAGE_INSTRUCTION; curate translations over time.
5. **Suggestion prompts** ŌĆö Landing page suggestions localized to actual translated questions since users send them verbatim. Backend handles multilingual input naturally.

## Key Decisions

- **Backend stays English**: Database, Qdrant vector index, and city API payloads are never translated. Translation happens at two boundaries only (synthesizer + frontend i18n). This avoids duplicating the entire retrieval layer per language.
- **Translation at two boundaries**: (1) The LLM synthesizer outputs in the user's language via appended LANGUAGE_INSTRUCTION, (2) React frontend localizes static UI strings via react-i18next with bundled JSON catalogs.
- **Forced synthesis for non-English follow-ups**: English-based `needs_synthesis()` heuristics (pronoun detection, follow-up patterns) won't match non-English text reliably. All non-English follow-ups bypass the heuristic and force conversation synthesis via Haiku.
- **Loose `str` type for language field** (not Literal enum): Future languages work without backend code changes ŌĆö just add locale files and a selector option.
- **DB schema v8** (not v7): v7 was taken by Stripe payment system's `tier`/`stripe_customer_id`/`stripe_subscription_id` columns.
